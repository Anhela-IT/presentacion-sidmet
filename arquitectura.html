Arquitectura del sistema

El portal de boletines meteorológicos se construye sobre Liferay 7 DXP, una plataforma empresarial Java EE que gestiona la autenticación SSO, el control de acceso por roles y el ensamblado de la interfaz mediante portlets.

La lógica de negocio se distribuye en 13 módulos OSGi independientes que se comunican con 6 microservicios backend a través del módulo transversal aemet-api-client, único punto de entrada a todos los servicios externos.

01 — Stack técnico

Capa de presentación — Tema + JSP

El tema boletines-theme define la estructura HTML global mediante plantillas Freemarker (portal_normal.ftl, navigation.ftl). Cada portlet renderiza sus propias vistas JSP con Liferay MVC. Los estilos siguen el sistema de diseño propio de AEMET: Nunito Sans, azul #1950a0 y naranja #faaf00, con arquitectura SCSS modular.

Las peticiones AJAX (filtros, paginación, carga de datos) se resuelven mediante MVCResourceCommand desde el mismo portlet, devolviendo JSON al frontend.

↓

Capa de negocio — Portlets OSGi (Liferay MVC)

Cada módulo es un bundle OSGi desplegado de forma independiente en el contenedor Felix de Liferay. Siguen el patrón Liferay MVC: un *Portlet.java como controlador principal, *MVCRenderCommand para las vistas, *MVCActionCommand para los formularios y *MVCResourceCommand para las llamadas AJAX/JSON.

El módulo boletines incluye además la librería iTextPDF como recurso embebido (@itextpdf-*.jar) para la generación de documentos PDF.

↓

Capa de integración — aemet-api-client

Bundle OSGi registrado como servicio (@Component … service = ApiGateway.class). Cualquier módulo que necesite datos externos lo inyecta mediante @Reference ApiGateway. Encapsula la configuración de los 6 endpoints (URLs, timeouts) a través de OSGi ConfigAdmin (AemetApiConfiguration), modificable en caliente desde el panel de control de Liferay sin reiniciar el servidor.

Soporta GET, POST JSON, PUT JSON, PATCH JSON y DELETE. Las respuestas se leen con soporte gzip y se devuelven como String JSON para que cada módulo deserialice según su modelo de dominio.

↓

Capa de datos — Microservicios backend

Seis microservicios REST independientes, cada uno con su propio puerto, exponen los datos al portal. El módulo aemet-api-client resuelve la URL correcta según el enum ApiBase (BULLETINS, CERTIFICATES, REPORTS, REPORTS_PDF, ADMINISTRACION, NOTIFICACIONES, ZONIFICACION).

02 — Microservicios backend
Bulletins :5003 — BULLETINS Boletines meteorológicos provinciales y marítimos (incluido NAVTEX). Fuente principal del módulo boletines.
Certificates :5001 — CERTIFICATES Gestión de certificados meteorológicos oficiales: solicitud, consulta, edición y borrado.
Reports :5002 — REPORTS Datos meteorológicos para informes personalizados: series temporales, parámetros y zonas.
Reports PDF :5004 — REPORTS_PDF / ZONIFICACION Generación de documentos PDF de informes y servicio de zonificación geográfica.
Administración :5005 — ADMINISTRACION Gestión de usuarios, reglas de negocio, logs de actividad e indicadores de uso.
Notificaciones :8002/v1 — NOTIFICACIONES Envío y consulta de notificaciones al usuario. Único servicio con prefijo de versión /v1 en la URL base.
03 — Contrato del ApiGateway

La interfaz ApiGateway expone un método por verbo HTTP. Cada uno tiene una variante que recibe el enum ApiBase (resuelve la URL base) y otra que acepta una URL absoluta para casos especiales.

GET

Consulta de recursos. Acepta ruta relativa, ApiBase + ruta o URL absoluta. Soporta respuesta gzip.

POST

Creación de recursos. Envía un cuerpo application/json. Acepta ruta relativa, ApiBase + ruta o URL absoluta.

PUT

Reemplazo completo de un recurso. Acepta ApiBase + ruta o URL absoluta.

PATCH

Actualización parcial. Usa java.net.http.HttpClient (Java 11) porque HttpURLConnection no soporta PATCH de forma nativa.

DELETE

Eliminación de un recurso. Acepta ApiBase + ruta o URL absoluta.

Configuración en caliente

Las URLs base y los timeouts (connectTimeoutMs, readTimeoutMs) se configuran desde Panel de control → Configuración del sistema → aemet-boletines sin necesidad de reiniciar. El método @Activate / @Modified activate() del impl recarga los valores en tiempo de ejecución.

04 — Módulos OSGi del portal

Cada módulo es un bundle OSGi con su propio ciclo de vida. Se despliegan, actualizan y detienen de forma independiente. Los módulos del panel de administración son portlets separados que Liferay ensambla en la misma página.

boletines
BoletinesPortlet

Dos portlets: BoletinesProvincialesPortlet y BoletinesMaritimosPortlet. Gestiona el listado, filtrado, detalle y CRUD de boletines. Incluye iTextPDF como recurso embebido.

certificados
CertificadosPortlet

Listado, detalle, solicitud, edición y borrado de certificados meteorológicos oficiales.

informes
InformesPortlet

Creación de informes personalizados con selección de parámetros meteorológicos, descarga PDF y gestión completa del ciclo de vida.

area-personal
AreaPersonalPortlet

Perfil del usuario autenticado: nombre, correo, rol y opciones de configuración de cuenta.

gestion-usuarios
GestionUsuariosPortlet

Alta, edición, activación/desactivación y borrado de usuarios del portal. Solo rol Administrador.

logs
LogsPortlet · ConfigChangesLogsPortlet

Dos portlets: registro cronológico de actividad del portal y listado de cambios de configuración.

indicadores-uso
IndicadoresUsoPortlet

Estadísticas de acceso y uso: sesiones, módulos más consultados y métricas de actividad.

reglas
ReglasPortlet · ReglasMaritimasPortlet

Gestión de reglas que condicionan el comportamiento del portal. Dos portlets: reglas generales y reglas marítimas.

notificaciones
NotificacionesPortlet

Centro de notificaciones: listado, detalle y gestión de avisos enviados a los usuarios.

administracion
ZonificacionPortlet

Portlet de zonificación geográfica dentro del panel de administración. Gestiona las zonas de predicción.

aemet-api-client
Export-Package: aemet.api.client

Bundle transversal. Exporta el paquete aemet.api.client para que todos los módulos puedan inyectar el servicio ApiGateway vía OSGi.

header
HeaderPortlet

Cabecera global del portal: menú de usuario, acceso al área personal y notificaciones activas.

boletines-login-redirect
login-redirect

Redirige al usuario tras el login a la página correcta según su rol, evitando la página de inicio por defecto de Liferay.

06 — Usuarios y roles

Los usuarios no se crean manualmente en el portal: se importan desde el LDAP corporativo. Una vez importados, un administrador los da de alta en el sistema asignándoles uno de los dos roles disponibles.
